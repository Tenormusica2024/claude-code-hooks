## hook_utils: transcript 読み取り失敗時の挙動（`load_transcript`）

**決定: 失敗時はスキップ（空リスト返却）**

transcript ファイルが読めない場合（権限エラー・ファイル消失・破損等）、`load_transcript` は空リスト `[]` を返してフックをそのまま通す。

- **block 方向にしない理由**: システム側の問題でフックが毎回誤 block するノイズが設計上の欠陥になる
- **レビューで複数回指摘されるパターン**: 再レビューで同じ指摘が出たら「意図的設計」として却下する

## document-update-detector: append 省略形の trigger_kind 分離（`doc_smart_append`）

**決定: `doc_smart` と `doc_smart_append` を別の trigger_kind として区別する**

「ドキュメントに追記して」（明示パスなし）を `doc_smart`（更新モード）に統合すると追記意図が失われ、70% 縮小・整合性チェックが適用されてしまう。

- **分離した理由**: append と update は注入するコンテキストの内容が異なる。畳み込みは実装コストが低いが正確性を損なう
- **再レビューで指摘されるパターン**: この分離を「重複」として統合を勧める指摘は却下する

---

## test-complete-hook: Bash 加点は +1（弱証拠として維持）

**決定: `has_bash` による加点を +2 → +1 に引き下げる**

Bash は git・pip・ファイルコピー等のテスト無関係な操作でも使われる。+2 のままだと他の証拠との組み合わせで閾値(6)に届きやすくなり誤発火率が上がる。

- **+1 に抑制した理由**: 「Bash を使った」は弱証拠として残す価値がある（テストフレームワーク未検出でも実行の痕跡）。ただし単独では閾値に届かない重みにする
- **条件付き加点（detect_test_command 一致時のみ）にしなかった理由**: +4（コマンド痕跡）と判定条件が重複し、独立した証拠としての意味がなくなる
- **再レビューで指摘されるパターン**: 「+2 に戻すべき」または「条件付きにすべき」の指摘は、上記トレードオフを無視しているため却下する

---

## merge_parallel_reviews: exit code の意味論（1=全失敗 / 2=部分失敗）

**決定: 非準拠モデルが存在する場合は exit code 2 で部分失敗を通知する**

一部モデルが非準拠出力を返しても残りが成功していれば結果は出力される。しかしその状態を呼び出し元に伝える手段がなければ「指摘が消えた」ことに気づけない。

- **stdout 警告ヘッダーにしなかった理由**: rfl の Markdown パーサーが警告を finding として誤解析するリスクがある。stdout は純粋な結果のみに保つ
- **exit code 体系**: exit 0 = 全成功、exit 1 = 全モデル失敗（エラー）、exit 2 = 部分失敗（一部指摘欠落の可能性あり）
- **再レビューで指摘されるパターン**: 「exit code 2 は非標準」という指摘は却下する。rfl 側でハンドリングすることが前提の設計

---

## document-update-detector: doc_smart の70%縮小は CLAUDE.md のみ適用

**決定: `rules/*.md` に対して70%縮小ターゲットを適用しない**

`doc_smart` の更新モードで `rules/*.md` が選択された場合、圧縮すると詳細仕様・設計決定の情報が削られすぎる。

- **CLAUDE.md との違い**: CLAUDE.md は「軽量インデックス」が設計目標なので縮小が有効。`rules/*.md` は詳細仕様の保管場所なので圧縮すると価値が下がる
- **レビューで指摘されるパターン**: 「全候補に統一ルールを適用すべき」という指摘は、ファイルの役割差を無視しているため却下する
